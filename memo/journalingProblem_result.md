# 🚀 비동기 크롤러 아키텍처 개선 및 장애 복구 설계 요약

## 📌 1. 핵심 문제: 메모리(RAM)의 휘발성과 단일 장애점(SPOF)
* **기존 방식의 약점:** 4,000건의 크롤링이 모두 끝날 때까지 
  데이터를 파이썬 메모리(리스트)에 들고 있다가 마지막에 한 번에 
  CSV로 저장하는 구조입니다.
* **치명적 리스크:** 99% 진행된 상태에서 VPN이 끊기거나 대상 
  서버가 응답하지 않으면, 메모리에 모아둔 모든 데이터가 강제 
  종료와 함께 허공으로 증발해 버리는 대참사가 발생합니다.

---

## 💡 2. 개선 방안: 체크포인팅(Checkpointing)과 즉시 저장(Append)
* **해결 원리:** 데이터가 1건 파싱 완료될 때마다 메모리에 
  쌓아두지 않고, 디스크에 있는 CSV 파일 맨 아랫줄에 즉시 
  이어붙여 씁니다(Append 모드). 
* **네트워크 I/O vs 디스크 I/O:** 초당 수천 건이 쏟아지는 채팅 
  서버라면 디스크 쓰기가 병목(Bottleneck)이 됩니다. 하지만 
  현재 크롤러는 대상 서버를 배려해 1~2초의 지연(Jitter)을 두고 
  통신합니다. 수십 마이크로초(µs)밖에 안 걸리는 디스크 쓰기 
  작업은 성능에 전혀 악영향을 주지 않습니다.
* **기대 효과:** 언제 컴퓨터 전원을 뽑아도, 마지막으로 수집된 
  말 데이터까지는 디스크에 100% 안전하게 보존됩니다. 다시 
  실행하면 중단된 지점부터 완벽하게 이어서 작업(Resume)할 수 
  있습니다.

## 🏛️ 3. 실무적 결단: 정렬(Sorting)에 대한 관심사의 분리
* **논리적 딜레마:** 비동기 통신은 먼저 처리되는 순서대로 응답이 
  옵니다(예: 1번 -> 3번 -> 2번). 이를 완료되는 즉시 CSV에 쓰면 
  파일의 물리적 순서도 자연스럽게 섞이게 됩니다.
* **최종 설계 방향 (A 방식 채택):** 크롤러의 코드를 복잡하게 
  만들면서까지 파일 저장 순서를 강제하지 않기로 결정했습니다. 
    * **크롤러의 역할:** 순서에 상관없이 데이터를 무조건 빠르고 
      안전하게 파일에 밀어 넣는 데만 100% 집중합니다.
    * **정렬의 역할:** 나중에 엑셀(Excel)이나 데이터베이스
      (MariaDB)의 `ORDER BY` 기능을 활용하여 원본 마번(HRNO) 
      순서대로 정렬합니다. 
* **결론:** 각 시스템(파이썬 크롤러 vs DB/엑셀)이 자신이 가장 
  잘하는 일만 수행하도록 책임을 나누는, 실무 표준에 부합하는 
  아주 깔끔한 아키텍처가 완성되었습니다.
