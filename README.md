# 🐎 Netkeiba 경마 데이터 비동기 크롤러 (Async Crawler Refactoring)

## 📌 프로젝트 개요
일본 경마 사이트(netkeiba.com)에서 특정 말(HRNO)의 상세 정보, 
혈통, 출주 성적을 수집하는 파이썬 기반 웹 크롤러입니다. 

초기에는 `requests` 라이브러리를 이용한 **동기식(Synchronous)** 구조였으나, 대량의 데이터 수집 시 발생하는 네트워크 대기 시간
(I/O 낭비)을 해결하기 위해 `asyncio`와 `aiohttp`를 도입하여 
**비동기식(Asynchronous) 코루틴 아키텍처**로 전면 리팩토링하였습니다.

---

## 💡 주요 개선 사항 및 컴퓨터 공학 원리

본 프로젝트는 다음과 같은 네트워크 및 운영체제 핵심 개념을 
고려하여 설계되었습니다. (배경지식이 필요할 경우 아래 키워드 
검색을 권장합니다.)

### 1. I/O 대기 시간 최소화 (Blocking vs Non-Blocking I/O)
* **문제점:** 기존 `requests` 방식은 서버에 HTML을 요청한 후 
  응답이 올 때까지 스레드 전체가 정지(Blocking)되어 자원이 
  낭비되었습니다.
* **개선 (원리):** 코루틴을 적용하여 네트워크 응답을 기다리는 
  동안 프로그램이 멈추지 않고(Non-Blocking), 다른 말의 페이지를 
  즉시 요청하도록 제어권을 넘깁니다. 
* **결과:** 단일 스레드(Single Thread)만으로도 수많은 HTTP 통신을 
  딜레이 없이 처리하여 전체 크롤링 속도가 비약적으로 상승했습니다.

### 2. 내부 AJAX 병렬 처리 (Concurrency)
* **문제점:** 말 한 마리의 데이터를 완성하려면 [메인 상세 페이지] 
  -> [혈통 AJAX] -> [성적 AJAX] 순서로 3번의 직렬 통신이 
  필요했습니다.
* **개선 (실무 고도화):** 메인 페이지 로드 후, 종속성이 없는 혈통 
  데이터와 성적 데이터 요청을 `asyncio.gather()`를 통해 **동시에
  (병렬로)** 서버에 쏘도록 최적화했습니다. 단일 말 데이터를 
  완성하는 데 걸리는 시간이 절반으로 단축되었습니다.

## 🚀 실무 고도화 단계: 서버 보호 및 차단 방지 아키텍처

비동기 크롤러는 속도가 매우 빠르기 때문에 대상 서버의 방어 
시스템(WAF, 웹 방화벽)에 디도스(DDoS)나 악성 봇(Bot)으로 
오인되어 IP가 차단(Rate Limiting)될 위험이 큽니다. 이를 
방지하기 위해 다음 두 가지 방어 기법을 적용했습니다.

### 1. 세마포어 (Semaphore)를 통한 동시성 통제
```python
# 동시에 최대 3개의 통신만 서버로 나갈 수 있도록 
# 톨게이트 역할을 하는 세마포어 적용
sem = asyncio.Semaphore(3)
```
100개의 파싱 작업이 코루틴 큐에 들어가더라도, 실제로 네트워크를 
타고 나가는 요청은 동시에 딱 3개로 엄격하게 유지됩니다. 이를 
통해 서버에 과부하를 주지 않는 아슬아슬한 한계선을 유지하며 
최고 효율을 뽑아냅니다.

### 2. 지터 (Jitter)를 활용한 인간화(Humanize) 딜레이
```python
# 요청 간격에 1.0초 ~ 2.0초 사이의 불규칙한 난수 대기 부여
delay = random.uniform(1.0, 2.0)
await asyncio.sleep(delay)
```
고정된 시간(예: 딱 1초)이 아니라 불규칙한 딜레이를 부여하여 
기계적인 패턴을 숨깁니다. 여기서 중요한 점은 `time.sleep`이 아닌 
`asyncio.sleep`을 사용하여, 대기 시간 동안에도 전체 파이프라인
(다른 코루틴)은 멈추지 않고 100% 가동된다는 점입니다.

* **Async Network:** `aiohttp`, `asyncio`
